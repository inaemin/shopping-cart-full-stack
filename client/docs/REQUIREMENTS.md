## 요구사항 정리 (주문 / 결제)

> step3에서 설계한 시스템 명세를 프론트엔드 관점으로 재해석한 것.
> 계산(주문금액/할인/배송비/결제금액)은 모두 서버(SSOT)가 책임지고, 프론트는 그 상태를 조회·동기화해서 화면에 반영한다.

- [ ] 장바구니에서 주문 확인으로 진입할 수 있다.
  - 장바구니에서 '주문 확인'을 누르면 주문 생성 API를 호출한다.
  - 성공 시 응답으로 받은 id로 주문 확인 페이지(`/checkout/:checkoutId`)로 이동한다.

- [ ] 주문 확인 페이지(`/checkout/:checkoutId`)를 렌더링한다.
  - 진입 시 주문 조회 API(`GET /checkouts/:checkoutId`)로 데이터를 받아 렌더링한다. (이 응답이 SSOT)
  - 상품 종류/개수, 상품 목록, 쿠폰 목록, 주문금액·쿠폰할인·배송비·총결제금액을 화면에 표시한다.
  - 새로고침해도 같은 페이지에 머문다. (상태를 클라이언트가 아닌 서버 데이터로부터 복원)
  - 페이지에서 이탈할 때 주문 취소 API(`DELETE /checkouts/:checkoutId`)를 호출한다. (언마운트 시)

- [ ] 배송 정보(제주/도서산간)를 변경하면 금액이 갱신된다.
  - 체크박스 토글 시 배송정보 수정 API(`PATCH /checkouts/:checkoutId/address`)를 호출한다.
  - 성공 후 SSOT를 invalidate → 재조회하여 배송비/총결제금액을 갱신한다.
  - 배송비 정책은 화면에서 안내 문구로 보여준다. (예: 10만원 이상 무료배송)

- [ ] 쿠폰 모달에서 쿠폰을 선택하고 적용할 수 있다.
  - '쿠폰 적용' 버튼으로 모달을 연다. (모달 열림 상태는 클라이언트 상태)
  - 진입 시 최대 혜택 조합이 기본 선택되어 있어야 한다.
  - 체크 여부에 따라 예상 할인액이 유동적으로 변하며, 적용 버튼에 표시된다.
    - 예상 할인액은 쿠폰 할인액 계산 API로 조회한다. (선택 쿠폰 조합이 바뀌면 자동 재요청)
  - 최대 2개까지 선택할 수 있다. (초과 선택 시 비활성화)
  - 선택할 수 없는 쿠폰은 비활성화 상태로 보여준다.
    - 최소주문금액 미달, 사용가능시간 미충족, BTGO 조건 미충족, 이미 무료배송 조건을 만족(무료배송 쿠폰) 등.
    - 선택 가능 여부 판단은 서버가 내려준 값(disabled)을 따른다.
  - '사용하기'를 누르면 쿠폰 적용 API(`PATCH /checkouts/:checkoutId/coupons`)를 호출하고 모달을 닫는다.
    - 성공 후 SSOT를 invalidate → 재조회하여 쿠폰할인/총결제금액을 갱신한다.
  - '사용하기'를 누르지 않고 모달을 닫으면 직전에 적용돼 있던 쿠폰 상태로 복구된다.
    - 모달을 조건부 렌더링(닫으면 unmount)하여, 다시 열 때 서버 데이터 기준으로 재초기화한다.

- [ ] 결제 후 결제 성공 페이지(`/order-success`)로 이동한다.
  - '결제하기'를 누르면 결제 정보 전송 API(`POST /checkouts/:checkoutId/payment`)를 호출한다.
  - 응답(종류/개수/최종 결제금액)을 navigate state로 넘기며, 뒤로가기로 돌아오지 못하게 replace로 이동한다.
  - 결제 성공 페이지는 주문 확인 페이지를 거쳐서만 접근 가능하다. (navigate state가 없으면 장바구니로 이동)

## ✅ 전체 프로그래밍 요구사항

> 이전 미션의 프로그래밍 요구사항은 기본으로 포함한다.

### 테스트

- 안정적인 소프트웨어를 만들기 위해서, 자동화 테스트 범위(E2E, 단위, 통합)와 대상을 스스로 정의하고, 테스트 한다.
- 테스트 대상 선정 이유를 PR 에 남긴다.

### 인터페이스

- 함수/클래스의 인터페이스는 복잡한 구현을 숨긴다.
- 호출부에서 알 필요 없는 것은 매개변수로 받지 않는다. 내부 구조를 알아야만 이해가 되는 매개변수를 조심한다.

### Library

- 스타일링에는 CSS Module, styled-components, emotion 중 한 가지를 선택하여 사용한다.
- 페이지 전환에 react-router 를 사용하여도 무방하다. (필수 아님)
- 이전에 학습하였거나(msw, storybook), 명시된 라이브러리 외에는 사용하지 않고 직접 구현한다.

## ✅ FE 프로그래밍 요구사항

### 컴포넌트 추상화

- 비슷한듯 다른 UI 를 구현할 때, 추상화 여부를 결정한다.
  - 추상화의 근거는 무엇일까? 고민한다.

### 상태 관리

- 서버 상태를 하나의 원천 상태(Single Source of Truth)로 관리해야 한다.
- 서버 응답과 클라이언트 상태의 책임을 명확히 나누어 관리할 수 있다.
- React의 상태 관리 훅(useState, useReducer)을 사용하여 상태를 구성하고, 필요 시에 Context API 등을 활용해 해당 상태가 필요한 컴포넌트 트리에 상태를 전달합니다.
- 결제 금액, 배송비 등의 파생 상태(Derived State)를 동적으로 계산한다.

### Data fetching hook

- 서버 API 통신 결과를 관리할 수 있는 함수를 직접 개발한다.
- 반환값에는 데이터, 로딩 여부, 에러 정보 등이 포함되어야 한다.
- 상용 라이브러리의 인터페이스를 모방하는건 무의미하다. 동작 매커니즘을 이해하고 직접 만들어보는게 훨씬 중요하다.
  - "어떻게 SSOT 지켰을까?", "어떻게 캐싱을 할까?", etc...

### 코드 퀄리티

- 컴포넌트와 비즈니스 로직을 분리하여, 재사용 가능하게 한다.
  - 재사용 가능하게 하기 위해서, 컴포넌트 내에서 함수를 호출하는 것과, props 를 가지는 방식을 구별하고, 더 나은 방법을 채택한다.
- 사용하기 좋은 인터페이스를 고민한다. 이 컴포넌트의 다양한 use case 를 떠올려본다.
- refetchCart, deleteCartItem 같이 세부 구현을 나타내는 props 가 있다면 이는 이름부터 안티 패턴이다. 다른 use case 에 대응이 어렵다. 1단계 서버 미션에서 다룬, DIP 를 떠올려보자.
